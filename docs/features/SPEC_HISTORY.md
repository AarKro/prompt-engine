# Feature Specification: History
## Overview
The History feature provides a persistent record of prompts generated by the user. It automatically captures the output whenever a copy action occurs, storing it locally for later retrieval. This feature ensures users can quickly reuse previous work without having to reconstruct the prompt from scratch.
## Data Management
Storage Mechanism

* Technology: Use the browser's localStorage API to ensure data persists across sessions.
* Trigger: A new entry is created immediately whenever the user copies a prompt (either via the "Copy" button or the Command + C shortcut in the Prompt Editor).
* Data Structure: Each history entry should store:

The full text of the compiled prompt.
(Optional but recommended) Timestamp for sorting or future display features.


* Capacity Limit: The history list must store a maximum of 50 prompts.
* Overflow Logic: When saving a 51st prompt, the oldest entry (first-in-first-out) must be removed to accommodate the new one.

## User Interface (UI) & Layout
### List View

* Layout: A vertical, top-to-bottom list.
* List Items:

Each entry is contained within a distinct "box" or card element.
Content: display the first 10 words of the saved prompt text.
Truncation: If the prompt is longer than 10 words, use an ellipsis (...) or a fade effect to indicate continuation.
Styling: The boxes should be clickable affordances (e.g., hover effects) to indicate interactivity.



## Functional Requirements & Interaction
### Saving History

* The system must detect a successful "Copy" event initiated in the Prompt Editor (see SPEC_PROMPT_EDITOR.md).
* Upon detection, serialize the current prompt content and save it to localStorage.

Restoring a Prompt (Click Behavior)
When a user clicks on a history item entry, the following two actions must occur in sequence:

1. Clipboard Copy: The full text of the selected prompt (not just the 10-word preview) is automatically copied to the user's clipboard.
2. Editor Population: The prompt text or state is loaded back into the Prompt Editor.

Note: If the editor architecture separates text into dynamic fields, the system should ideally attempt to populate these fields. If only the raw text string is stored in history, the editor should display this full text or populate the active template with this content as best as possible.



### Dependencies

* This feature relies on the output of the Prompt Editor. The "Copy" events within the Prompt Editor serve as the trigger for saving history items.
